# Data Bridge v0.2.1 — Korean-aware hybrid retrieval (trigram fusion)

> Status: **complete** — merged (PR #3); live golden gate passed 2026-07-22
> (`keyword_hit=1.000`, `source_hit=5/5`, no regression). The §4 recommendation
> (Korean golden items) and the §5 production note (prod `pg_trgm`) are closed in
> [v0.2.2](v0.2.2.md).
> Scope: closes the Korean full-text-search limitation carried forward from
> [v0.2.0 §6](v0.2.0.md).

## 1. What changed

The v0.2.0 hybrid search fused two ranked candidate lists — vector (cosine) and English
full-text (`to_tsvector('english', …)`) — via reciprocal rank fusion. English FTS gave
weak Korean keyword recall: Korean is agglutinative, so `배포를` / `배포가` / `배포는`
tokenize as *distinct* words under the stock parser. A bare query term `배포` shares no
token with any of them, so the FTS path returns nothing for those queries. Vector recall
(multilingual embeddings) still worked, which is why Korean Q&A functioned end-to-end —
but the keyword signal was effectively English-only.

v0.2.1 adds a **third RRF candidate source: `pg_trgm` character-trigram similarity** over
raw content, using `word_similarity(query, content)` (the query's trigrams matched against
the most similar contiguous extent of a chunk — the right metric for a short query against
a longer chunk). Character trigrams are language-agnostic: they match `배포` inside
`배포를` without any morphological analyzer. The English `content_tsv` path is untouched,
so English keyword quality is preserved; the trigram source runs alongside it and RRF fuses
whichever sources produced candidates.

```
v0.2.0:  RRF( vector , english-FTS )
v0.2.1:  RRF( vector , english-FTS , trigram )   ← trigram recovers Korean keyword recall
```

## 2. Design decision — pg_trgm, not pg_bigm (portability, design D-3)

The Korean-recall fix had to hold the line on **D-3: "runs identically on the pgvector
container, Cloud SQL, and AlloyDB."** Options considered:

| Option | Korean recall | Portability (D-3) | Verdict |
|---|---|---|---|
| `simple` text-search config | ~none (doesn't segment 조사) | ✅ | rejected — doesn't fix the real cause |
| **`pg_bigm`** (bigram FTS) | best for CJK | ❌ **not in `pgvector/pgvector:pg16`** | **rejected — breaks D-3** |
| **`pg_trgm`** (trigram) | good (n-gram sidesteps morphology) | ✅ stock contrib, present locally + Cloud SQL + AlloyDB | **chosen** |

`pg_bigm` availability was checked against the live local image and is **absent**; adopting
it would require a custom Postgres image and forfeit the "interchangeable behind one
connection string" guarantee. `pg_trgm` is stock contrib and verified present
(`pg_available_extensions`). Bigram is theoretically stronger for Korean than trigram; that
marginal edge was traded for portability — the deliberate call recorded here.

## 3. Implementation

- **`store/schema.sql`** — `CREATE EXTENSION IF NOT EXISTS pg_trgm;` + a `GIN (content
  gin_trgm_ops)` index. The `content_tsv` english column/index are unchanged.
- **`store/pg.py`** — `search_hybrid` gains a trigram candidate query
  (`%(query_text)s <% content`, threshold via `set_config('pg_trgm.word_similarity_threshold', …)`),
  a new `trgm_threshold` parameter (default `0.2`, validated to `[0, 1]`), and a
  three-source RRF loop. `SearchHit` gains `trgm_rank` for trace observability. The
  trigram source degrades gracefully to empty when nothing clears the threshold, exactly
  like the existing FTS path.

## 5. Known limitations (updated from v0.2.0 §6)

- **Resolved:** "Postgres FTS uses `'english'` … no Korean stemming" — Korean keyword recall
  is now provided by the trigram source.
- **New / accepted:** trigram is language-agnostic but coarser than a true morphological
  analyzer; `trgm_threshold` (default `0.2`) trades recall vs. noise and is tunable per
  deployment. RRF de-weights low-ranked trigram matches, so noise is bounded.
- **Production note:** first prod deploy needs `CREATE EXTENSION pg_trgm` privilege on the
  target Cloud SQL / AlloyDB instance (owner-verify).
- All other v0.2.0 §6 limitations carry forward unchanged.
